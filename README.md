## Задание 1. СУБД
Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

Приведите ответ в свободной форме.

1.1. Для бюджетирования и финансовой аналитики оптимально подойдёт реляционная СУБД (например, PostgreSQL, Oracle, Microsoft SQL Server).

Обоснование:

гарантирует целостность данных через транзакции (ACID);

поддерживает чёткую структуру данных с таблицами, связями и ограничениями;

позволяет строить сложные аналитические запросы с JOIN, агрегациями;

имеет развитые инструменты для отчётности и интеграции с BI‑системами (Tableau, Power BI).

1.1*. Для ускорения хеширования можно использовать:

криптографические библиотеки с аппаратным ускорением: OpenSSL (с поддержкой Intel SHA extensions), BoringSSL;

специализированные API языков программирования: в Python — модуль hashlib с алгоритмами SHA‑256/512; в Java — класс MessageDigest;

вынесение хеширования на GPU через OpenCL или CUDA (для массовых операций);

кэширование результатов хеширования (например, через Redis).

1.2.

Для лендингов — NoSQL‑документная СУБД (MongoDB, Couchbase). Быстрое прототипирование, гибкая схема для разных форм лидов, высокая скорость записи данных с веб‑страниц.

Для CRM — реляционная или гибридная СУБД (PostgreSQL с JSONB, MySQL). Нужна структура для связей «клиент — сделка — менеджер», отчётность, история взаимодействий.

1.2*. Да, можно использовать PostgreSQL с расширением JSONB:

для лендингов — хранить лиды как JSON‑документы (гибкость);

для CRM — использовать реляционные таблицы с внешними ключами (целостность);

единый сервер снижает затраты на администрирование.

1.3. Оптимально — иерархическая или документоориентированная СУБД (например, MongoDB или XML‑база).

Обоснование:

данные естественно ложатся в древовидную структуру (отдел → подразделение → норматив → материал);

простой поиск по иерархии;

удобство редактирования и версионирования документов.

1.3*. Да, можно использовать MongoDB из задачи 1.2:

создать коллекцию «Корпоративные нормы» с полями: department, subunit, document, version, approval_date;

настроить права доступа по отделам через ролевую модель MongoDB;

интегрировать с корпоративным порталом через REST API.

1.4. Лучший выбор — графовая СУБД (Neo4j, Amazon Neptune).

Обоснование:

маршруты и курьеры — это сеть узлов (объекты, склады) и рёбер (дороги, временные интервалы);

графовые запросы (например, алгоритм Дейкстры) находят оптимальные пути за доли секунды;

легко учитывать динамические ограничения (пробки, приоритет доставки).

1.4*. Можно подключить отдел закупок к той же СУБД:

добавить узлы «поставщик», «заказ», «товар»;

связать с маршрутами: «заказ → склад → маршрут доставки»;

использовать общие данные о складах и транспорте;

отдельная СУБД нужна только при жёстких требованиях к изоляции данных закупок.

1.5*. Теоретически можно, но с оговорками. Оптимальный вариант — мультимодельная СУБД (ArangoDB, OrientDB):

графовая модель — для логистики;

документоориентированная — для лендингов и норм;

реляционная — для бюджетирования (через эмуляцию связей);

риски: сложность администрирования, возможная потеря производительности в узких местах.



## Задание 2. Транзакции
2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий.

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

Приведите ответ в свободной форме.

2.1. Пополнение баланса телефона (6 шагов):

Начало транзакции. Система блокирует счёт пользователя и баланс телефона для изменений.

Проверка условий. Проверяется: достаточно ли средств на счёте, корректен ли номер телефона, не превышен ли лимит пополнения.

Списание средств. Сумма списывается со счёта пользователя, фиксируется в журнале операций.

Зачисление на баланс. Эквивалентная сумма зачисляется на баланс телефона, обновляется статус номера.

Фиксация транзакции. Изменения сохраняются в БД (команда COMMIT).

Уведомление. Пользователь получает SMS/push: «Пополнение на 500 руб. успешно. Баланс: 1 200 руб.».

2.1*. При автоплатеже добавляются шаги:

перед шагом 1 — проверка расписания (наступила ли дата автоплатежа);

после шага 5 — обновление настроек автоплатежа (дата следующего платежа, статус «выполнено»).

Если на счёте недостаточно средств:

транзакция откатывается (ROLLBACK);

пользователь получает уведомление: «Автоплатёж отложен: недостаточно средств».

## Задание 3. SQL vs NoSQL
3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

Приведите ответ в свободной форме.

3.1. Преимущества SQL перед NoSQL:

Целостность данных (ACID). Гарантирует надёжность транзакций: все операции либо выполняются полностью, либо отменяются.

Чёткая схема данных. Жёсткая структура таблиц упрощает проектирование и поддержку сложных систем.

Мощные запросы. SQL поддерживает JOIN, подзапросы, оконные функции — идеально для аналитики.

Стандартизация. Единый язык запросов (ANSI SQL) — меньше зависимости от вендора.

Зрелые инструменты. Богатая экосистема: ORM (Hibernate), BI‑системы, ETL‑решения.

3.1*. Преимущества NewSQL:

Масштабируемость как у NoSQL (горизонтальное распределение данных);

ACID‑транзакции как у SQL (надёжность банковских и финансовых систем);

Высокая производительность за счёт in‑memory вычислений и оптимизированных движков (например, CockroachDB, VoltDB);

Гибкость схемы (частично): поддержка JSON‑полей без потери реляционной структуры;

Автоматическое шардирование — упрощение администрирования при росте данных.

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу выделено 1000 машин.

На основе какого критерия будете выбирать тип СУБД и какая модель распределённых вычислений здесь справится лучше всего и почему?

Оптимальная модель распределённых вычислений: MapReduce + распределённое хранилище (например, Hadoop HDFS + Spark):

Обоснование:

MapReduce разбивает вычисления на параллельные задачи (1 000 машин работают одновременно);

HDFS хранит данные с репликацией — устойчивость к сбоям узлов;
