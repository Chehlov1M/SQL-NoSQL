## Задание 1. СУБД

### Кейс

Крупная строительная компания (проектирование и девелопмент) создаёт архитектуру для работы с данными. Для каждой предметной области нужно подобрать оптимальный тип СУБД.

### Вопросы и ответы

#### 1.1. Бюджетирование проектов и финансовая аналитика

**Задача:** бюджетирование с формированием отчётов и прогнозированием рисков. Требуется целостность и чёткая структура данных.

**Решение:** реляционная СУБД (PostgreSQL, Oracle, Microsoft SQL Server).

**Обоснование:**
* целостность данных через транзакции (ACID);
* чёткая структура с таблицами, связями и ограничениями;
* поддержка сложных аналитических запросов (JOIN, агрегации);
* интеграция с BI‑системами (Tableau, Power BI).

#### 1.1*. Ускорение хеширования

**Проблема:** хеширование занимает много времени.

**Решения:**
* криптографические библиотеки с аппаратным ускорением: OpenSSL (Intel SHA extensions), BoringSSL;
* API языков программирования:
  * Python — модуль `hashlib` (SHA‑256/512);
  * Java — класс `MessageDigest`;
* вынесение хеширования на GPU (OpenCL, CUDA);
* кэширование результатов (например, через Redis).

#### 1.2. Лендинги и CRM

**Задача:** для каждого девелоперского проекта создаётся лендинг, данные по лидам стекаются в CRM. Требуется гибкость и скорость.

**Решение:**
* **для лендингов** — NoSQL‑документная СУБД (MongoDB, Couchbase): быстрое прототипирование, гибкая схема, высокая скорость записи;
* **для CRM** — реляционная или гибридная СУБД (PostgreSQL с JSONB, MySQL): структура для связей «клиент — сделка — менеджер», отчётность, история взаимодействий.

#### 1.2*. Одна СУБД для лендингов и CRM?

**Ответ:** да, PostgreSQL с расширением JSONB.

**Реализация:**
* лендинги: хранение лидов как JSON‑документов (гибкость);
* CRM: реляционные таблицы с внешними ключами (целостность);
* единый сервер — снижение затрат на администрирование.

#### 1.3. Корпоративные нормы и обучающие материалы

**Задача:** база по нормам, правилам и обучающим материалам с учётом структуры компании. Требуется простая и понятная структура.

**Решение:** иерархическая или документоориентированная СУБД (MongoDB, XML‑база).

**Обоснование:**
* древовидная структура (отдел → подразделение → норматив → материал);
* простой поиск по иерархии;
* удобство редактирования и версионирования документов.

#### 1.3*. Использование существующей СУБД

**Ответ:** да, можно использовать MongoDB из задачи 1.2.

**Реализация:**
1. Создать коллекцию «Корпоративные нормы» с полями:
   * `department`;
   * `subunit`;
   * `document`;
   * `version`;
   * `approval_date`.
2. Настроить права доступа по отделам через ролевую модель MongoDB.
3. Интегрировать с корпоративным порталом через REST API.

#### 1.4. Логистика: маршруты и курьеры

**Задача:** быстрое формирование маршрутов доставки и распределение курьеров. Требуется быстрая работа со связями.

**Решение:** графовая СУБД (Neo4j, Amazon Neptune).

**Обоснование:**
* маршруты и курьеры — сеть узлов (объекты, склады) и рёбер (дороги, временные интервалы);
* графовые запросы (алгоритм Дейкстры) находят оптимальные пути за доли секунды;
* учёт динамических ограничений (пробки, приоритет доставки).

#### 1.4*. Подключение отдела закупок

**Ответ:** можно подключить к той же СУБД.

**Реализация:**
* добавить узлы «поставщик», «заказ», «товар»;
* связать с маршрутами: «заказ → склад → маршрут доставки»;
* использовать общие данные о складах и транспорте;
* отдельная СУБД — только при жёстких требованиях к изоляции данных закупок.

#### 1.5*. Одна СУБД для всех задач?

**Ответ:** теоретически возможно, но с оговорками.

**Решение:** мультимодельная СУБД (ArangoDB, OrientDB).

**Реализация:**
* графовая модель — для логистики;
* документоориентированная — для лендингов и норм;
* реляционная — для бюджетирования (через эмуляцию связей).

**Риски:** сложность администрирования, возможная потеря производительности в узких местах.

### 2.1*. Пополнение через автоплатёж

Процесс аналогичен ручному пополнению, но включает дополнительные шаги автоматизации.

#### Дополнительные шаги:

**До шага 1 (инициирование):**
* **Проверка расписания автоплатежа.** Система банка (или платёжный сервис) проверяет, наступила ли дата/условие для автоплатежа (например, баланс < *** руб.).
* **Предварительная проверка.** Банк убеждается, что счёт активен, не заблокирован, и лимиты автоплатежа не превышены.

**После шага 5 (подтверждение транзакции):**
* **Обновление настроек автоплатежа.** В БД банка обновляются параметры автоплатежа:
   * дата следующего платежа (или условие);
   * статус «Выполнено»;
   * при необходимости — корректировка суммы (если платёж был динамическим).

#### Сценарий ошибки (недостаточно средств на банковском счёте)

* Банк не может выполнить списание — транзакция в банковской системе откатывается (`ROLLBACK`).
* Оператор связи не получает подтверждения платежа и не зачисляет средства.
* Клиент получает уведомления:
   * от банка: «Автоплатёж отложен: недостаточно средств на счёте»;
   * (опционально) от оператора связи: «Пополнение не выполнено. Проверьте баланс счёта».

#### Ключевые особенности автоплатежа

* автоматическая инициация по расписанию/условию;
* необходимость хранения и обновления настроек автоплатежа в БД банка;
* обработка сценариев частичной/неудачной оплаты.

#### Архитектурные особенности

* **Изоляция данных.** Банк управляет финансовыми операциями, оператор — балансом услуг. Системы взаимодействуют через защищённые API (REST, SOAP) или очереди сообщений (RabbitMQ, Kafka).
* **Согласованность.** Для обеспечения целостности используется двухфазная фиксация (2PC) или паттерн Saga.
* **Идемпотентность.** `transaction_id` гарантирует, что платёж не будет обработан дважды.
* **Логирование.** Каждая система ведёт журнал транзакций для аудита и разрешения споров.

Доработанное описание транзакции пополнения баланса телефона.
### Целевая транзакция охватывает весь процесс от инициирования платежа до финального подтверждения — и является неделимой: либо выполняется целиком, либо отменяется полностью (с откатом всех промежуточных изменений).
Этапы транзакции.
Инициирование и проверки в банке.
Клиент инициирует платёж на пополнение баланса телефона.
Банк выполняет проверки:
достаточность средств на счёте;
корректность реквизитов (номер телефона);
соблюдение лимитов.
При успешном прохождении проверок банк блокирует сумму на счёте клиента.
Фиксация операции и генерация transaction_id.
В БД банка создаётся запись о предстоящей операции списания.
Генерируется уникальный идентификатор транзакции — transaction_id.
Уведомление оператора связи.
Банк отправляет оператору связи сообщение о платеже, содержащее:
transaction_id;
номер телефона;
сумму платежа;
временную метку.
Проверка и зачисление у оператора связи.
Оператор связи получает запрос и проверяет его валидность, в т. ч. по transaction_id (чтобы исключить дублирование).
В БД оператора связи выполняется операция зачисления эквивалентной суммы на баланс указанного номера телефона.
Обновляется статус номера (например, снимается блокировка услуг при нулевом балансе).
Подтверждение транзакции.
Оператор связи отправляет банку подтверждение успешного зачисления средств.
Банк выполняет окончательное списание заблокированной суммы со счёта клиента.
В БД банка и БД оператора связи фиксируется статус «Завершено» для transaction_id.

Уведомление клиента.
Пользователь получает уведомления:
от банка: «Списано *** руб. Назначение: пополнение баланса телефона +X XXX XXX‑XX‑XX»;
от оператора связи: «Пополнение на *** руб. успешно. Баланс: *** руб.».

Обоснование соответствия определению транзакции.
Описанный процесс соответствует определению транзакции по следующим критериям:

Атомарность (неделимость).
Транзакция либо выполняется полностью (все этапы от 1 до 6), либо не выполняется вовсе.
Если на любом этапе возникает ошибка (например, оператор связи не подтверждает зачисление), происходит откат:
банк снимает блокировку суммы со счёта клиента;
в БД банка и оператора связи фиксируется статус «Отменено» или «Ошибка» для transaction_id;
никакие промежуточные изменения не сохраняются.

Согласованность (consistency).
Система переходит из одного согласованного состояния в другое:
До транзакции: баланс счёта клиента в банке — B1, баланс телефона у оператора — P1.
После успешной транзакции: баланс счёта клиента — B1−S, баланс телефона — P1 +S (где S — сумма платежа).
При отмене транзакции: балансы остаются неизменными (B1 и P1).

Изолированность (isolation).
Во время выполнения транзакции промежуточные состояния (например, заблокированная сумма в банке или зачисление на баланс телефона до подтверждения) не видны другим операциям.
Это предотвращает конфликты (например, двойное списание или использование незавершённого зачисления).
Долговечность (durability).
После фиксации статуса «Завершено» все изменения сохраняются в БД банка и оператора связи.
Даже в случае сбоя системы после подтверждения транзакция остаётся выполненной, а данные — доступными.

## Задание 3. SQL vs NoSQL

### 3.1. Преимущества SQL перед NoSQL

1. **Целостность данных (ACID).** Надёжность транзакций: все операции либо выполняются полностью, либо отменяются.
2. **Чёткая схема данных.** Жёсткая структура таблиц упрощает проектирование и поддержку сложных систем.
3. **Мощные запросы.** Поддержка JOIN, подзапросов, оконных функций — идеально для аналитики.
4. **Стандартизация.** Единый язык запросов (ANSI SQL) — меньше зависимости от вендора.
5. **Зрелые инструменты.** Богатая экосистема: ORM (Hibernate), BI‑системы, ETL‑решения.

### 3.1*. Преимущества NewSQL перед SQL и NoSQL

* **Масштабируемость** как у NoSQL (горизонтальное распределение данных);
* **ACID‑транзакции** как у SQL (надёжность для финансовых систем);
* **Высокая производительность** за счёт in‑memory вычислений (CockroachDB, VoltDB);
* **Гибкость схемы** (частично): поддержка JSON‑полей без потери реляционной структуры;
* **Автоматическое шардирование** — упрощение администрирования при росте данных.

## Задание 4. Кластеры

### Задача

Большое количество вычислений на 1 000 машинах с огромными объёмами данных.

### Решение

**Критерий выбора СУБД:** тип нагрузки и характер данных.

**Оптимальная модель:** MapReduce + распределённое хранилище (Hadoop HDFS + Spark).

**Обоснование:**
* **MapReduce** разбивает вычисления на параллельные задачи (1 000 машин работают одновременно);
* **HDFS** хранит данные с репликацией — устойчивость к сбоям узлов;
* **Spark** ускоряет обработку за счёт in‑memory вычислений (в 10–100 раз быстрее Hadoop MapReduce для итеративных алгоритмов);
* **гибкость данных** — поддержка структурированных, полуструктурированных и неструктурированных данных.

### Альтернативы

* для транзакционных нагрузок — NewSQL‑кластеры (CockroachDB);
* для аналитики в реальном времени — колоночные СУБД (ClickHouse, Vertica) с шардированием.
